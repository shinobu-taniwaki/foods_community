**最終更新**: 2026-05-26
**対象**: admin（しのぶさん）
**プロダクト**: マーケティングCampコミュニティ（MCC）
**前提資料**: `.claude/plans/foods-community-design.plan.md`（特に §5, §8, §13, §18）

## このマニュアルの使い方

- このドキュメントは、しのぶさんが手元で参照しながら β期間を運用するための実践マニュアルです。
- 各セクションのチェックリストはそのまま使えるよう設計されています。
- 不明点やトラブル発生時は、最後のセクション「困ったときの連絡先」を参照してください。
- β期間の全体像が知りたい場合は、まず §1 と §4「日々の運用フロー」を読めば概略がつかめます。

## 1. β期間の目的

β期間は、本ローンチ前にサービスを少人数で検証する期間です。以下5点を目的とします。

### 1.1 本ローンチ前のサービス検証
- 招待〜登録〜投稿〜退会の全フローが、実利用環境（実機・実回線）で支障なく動作するかを確認する。
- 50代中心・ITリテラシー低めの実ユーザーが「迷わず使えるか」を観察する。

### 1.2 機能の使い勝手フィードバック収集
- ボトムナビ、投稿UI、画像アップロード、PWAインストールなど、画面ごとの使い勝手を率直に聞き出す。
- 「使いにくい・わからない・気が引ける」が出た箇所をすべて記録する。

### 1.3 バグの洗い出し
- 限られた人数でも、開発時には見つからなかった不具合（端末依存・回線依存・操作順序依存）が必ず出る。
- 不具合報告フォームに集約し、即時対応 or v0.2 送りを切り分ける。

### 1.4 初期メンバーとの関係構築
- 「β初期メンバー」として、本ローンチ後も中心となるユーザーを育てる。
- しのぶさん自身がコメントで積極的に絡み、コミュニティの空気をつくる。

### 1.5 課金導線の検証（外出し運用が機能するか）
- プラン変更フォーム（Google フォーム）→ 振込確認 → admin 手動付与、という MVP の「外出し運用」が現場で回るかを検証する。
- 課金そのものは無料だが、本契約の「申し込み体験」を一部メンバーに試してもらう（任意）。

## 2. β期間の概要

| 項目 | 内容 |
|---|---|
| 期間目安 | 2 週間〜1 ヶ月（推奨：3 週間） |
| 対象人数 | 5〜10 名（クローズドβ） |
| 招待方法 | しのぶさんが個別に声掛け → 招待メール送信 |
| 招待時プラン | **`standard`** で発行（trial 制限を受けないため） |
| 課金 | **無料**（β中は請求しない／個別に伝達） |
| 終了告知 | 終了 1 週間前に admin_broadcast で全体通知 |
| 終了時処置 | admin が個別にプラン維持・降格・退会処理 |
| バナー | ヘッダー直下に「βテスト期間中」を期間中のみ表示 |

### 2.1 なぜ standard プランで招待するのか
- trial は機能制限がかかる（投稿・コメント不可、データ記録不可、運営からのアドバイス閲覧不可）。
- β期間は「全機能を試してもらってフィードバックを得る」のが目的なので、制限がないプランで招待する必要がある。
- premium はアプリ外特典が中心のため、アプリ機能検証目的では standard で十分。

### 2.2 期間の決め方
- 最短 2 週間：致命バグの洗い出しのみ。
- 推奨 3 週間：1 回は月次データ入力サイクルを回せる長さ。
- 最長 1 ヶ月：超えると初期メンバーの熱量が落ちやすい。

  - [ ] PWA でホーム画面に追加する方法
- [ ] **FAQ 5件**（`contents`／カテゴリ：column）
  - [ ] ログインできないときは
  - [ ] 画像がアップロードできないときは
  - [ ] 通知が来ない／多すぎるときは
  - [ ] 退会したいときは
  - [ ] 料金・プラン変更について

### 3.3 メンバー選定

- [ ] β初期メンバー候補リストアップ（5〜10名）
- [ ] 選定基準（以下 4 つを総合判断）
  - [ ] **IT リテラシー**：スマホで日常的に LINE・メール・Google フォームを操作できる
  - [ ] **コミュニケーション能力**：率直にフィードバックを出してくれる、わからないときに「わからない」と言える
  - [ ] **コミュニティへの関心**：他メンバーと交流する意欲がある
  - [ ] **業種バランス**：販売ジャンル（野菜・果物・米穀・水産・畜産・パン・乳製品・茶飲料・調味料・酒・乾物・その他）が偏らないよう 2〜3 ジャンルから選定
- [ ] 各メンバーに事前連絡（電話 or LINE）
  - 連絡内容：β招待の趣旨、無料であること、フィードバックを率直にお願いしたいこと、招待メールがいつ届くか
  - **メール送信前に必ず一報を入れる**（迷惑メール扱い回避と心理的安心のため）

### 3.4 admin 側の運用準備

- [ ] しのぶさんのプロフィールを完璧に整える（オーナーヘッダーに表示される）
  - [ ] アバター画像
  - [ ] 自己紹介
  - [ ] 屋号・地域・扱う商品
- [ ] admin 自身のテスト投稿を 1〜2 件投入（β招待発行前に削除 or 残すかは任意）
- [ ] Google フォーム回答の通知先メールを、しのぶさんが日常的にチェックする受信箱に設定

## 4. β期間中の運用フロー

### 4.1 招待発行

```
1. /admin/invites/new を開く
2. メールアドレス入力（β初期メンバー候補のメール）
3. プラン：standard を選択   ←【重要】trial にしない
4. [招待を送る] をクリック
5. invitations テーブルに INSERT、Resend で招待メール送信
6. 個別フォローアップ
   - 電話 or LINE で「招待メールを送りました。届かなければ迷惑メールフォルダを確認してください」と一報
   - リンク有効期限 7 日を伝える
```

#### 招待発行時の注意
- 同一メールに既に有効な招待がある場合は、二重発行せず先のトークンを再送する。
- 招待発行のたびに `audit_logs` に `invitation_created` が記録される（後で振り返り用）。

### 4.2 日々の運用（毎日チェック）

朝 1 回、夕方 1 回の合計 2 回、`/admin` ダッシュボードを開く。

- [ ] **未処理タスク**カードを確認
  - 招待未受諾 N 件 → 3 日以上経過していれば LINE で一報
  - 一時停止中 N 人（β中は基本的に発生しない想定）
- [ ] **新規投稿**を確認
  - 不適切な投稿（差別・誹謗中傷・宣伝のみ）がないか
  - admin がコメントで反応できるか（後述 §4.5）
- [ ] **新規コメント**を確認
  - メンバー間でトラブルの芽がないか
- [ ] **Sentry**にエラーが上がっていないか確認（朝 1 回でよい）
- [ ] **Plausible**で前日のアクティブユーザー数を確認

### 4.3 週次の運用（週 1 回／たとえば月曜朝）

- [ ] β期間バナーが正しく表示されているか、自分のスマホ実機で確認
- [ ] 未受諾の招待を確認
  - 1 週間経過したら本人に LINE で連絡、希望があれば再送 or 取消
- [ ] 投稿モデレーション
  - 投稿一覧（`/admin/posts`）を上から流し見し、内容を把握
  - 必要に応じて編集・削除（理由を入力、本人通知）
- [ ] コメントモデレーション（`/admin/comments`）
- [ ] お問い合わせフォーム・不具合報告フォームの回答を確認
- [ ] 監査ログ（`/admin/audit-log`）を週 1 で軽く流し見（自分の操作履歴の確認）

### 4.4 フィードバック収集

| 種別 | 収集方法 | 確認頻度 | 対応 |
|---|---|---|---|
| 機能要望・感想 | お問い合わせフォーム（Google フォーム） | 週次 | エクセル等に整理、v0.2 検討材料に |
| 不具合報告 | 不具合報告フォーム（Google フォーム） | **即時**（毎日確認） | 再現確認 → Claude Code に修正依頼 |
| 個別ヒアリング | 電話 or オンライン | **月 1 回／メンバーごと** | メモを残し、傾向を把握 |
| 観察 | Plausible・Sentry・投稿状況 | 週次 | 「使われていない機能」「離脱箇所」を発見 |

#### 個別ヒアリングの質問例（15〜20分）
1. 今週、コミュニティを開いた回数はどれくらいですか？
2. 一番よく見ているタブは？ 逆にほぼ見ていないタブは？
3. 投稿するときに「面倒だな」「迷ったな」と思った瞬間はありましたか？
4. 他のメンバーの投稿を見て、参考になったものはありましたか？
5. もし今すぐ 1 つだけ改善できるなら、どこを直してほしいですか？
6. ご家族・知人にこのコミュニティをすすめたいと思いますか？（理由も）

### 4.5 コンテンツ運用

- [ ] **週 2〜3 回**：お知らせ発信
  - news：業界トレンド、メンバーの活動紹介、運営からの一言
  - column：ノウハウ記事（短くてよい、500〜1500 文字目安）
  - seminar：必要時のみ
- [ ] **メンバーの投稿には可能な限り admin がコメントで反応**
  - 「コメント返ってきた」体験が継続率に直結する
  - 1〜2 行の短いコメントでよい（無理に深いコメントは不要）
- [ ] **「運営からのアドバイス」チャンネル**で個別事業者向けの助言
  - 投稿は admin のみ可能
  - 特定の事業者を念頭に置いた助言を、他メンバーにも参考になる形で書く

## 5. β期間終了時の運用

### 5.1 終了 1 週間前

- [ ] 全体通知（`/admin/broadcasts`）で「β期間終了のご案内」を送信
  - **メール並走を ON にする**（重要通知なので確実に届ける）
  - 件名・本文の構成例：

```
件名：【MCC】β期間終了のご案内（◯月◯日まで）

タイトル：β期間終了のお知らせ

本文：
皆さま、いつもMCCをご利用いただきありがとうございます。

このたびのβテスト期間は、◯月◯日（◯）をもって終了となります。
皆さまの率直なフィードバックのおかげで、サービスを大きく改善できました。
心より感謝申し上げます。

【今後について】
- 継続をご希望の方は、特にお手続きは不要です。
  運営側でスタンダードプランに切り替えさせていただきます。
  ※β期間終了後の料金については個別にご案内します。

- 退会をご希望の方は、お手数ですが
  「設定 → 退会申請」のフォームよりご連絡ください。

- プラン変更（プレミアム等）をご希望の方は
  「設定 → プラン情報」のフォームよりご連絡ください。

ご不明点はお問い合わせフォームまでお気軽にどうぞ。
これからもMCCをよろしくお願いいたします。

しのぶ
```

- [ ] 1 週間前のタイミングで、各メンバーに**個別に電話 or LINE**で同じ内容を一報する（メールだけだと埋もれる）

### 5.2 終了当日

事前に各メンバーの意向（継続／様子見／退会）を把握しておき、以下を一気に処理する。

- [ ] `/admin/members` を開き、メンバーごとに以下のいずれかを実行
  - **継続希望**：プランを **`standard`** に維持（既に standard なので操作不要、または premium に変更）
  - **様子見**：プランを **`trial`** に降格
    - 機能制限がかかる（投稿・コメント不可、データ記録不可）
    - 本人に通知が飛ぶ
  - **退会希望**：「退会させる」を実行
    - `profiles.status = 'deleted'`
    - 既存セッション失効
    - 投稿は admin のみ閲覧可能になる
- [ ] 環境変数 `NEXT_PUBLIC_BETA_MODE=false` に変更してデプロイ
  - βバナーが非表示になる
  - 念のため `NEXT_PUBLIC_BETA_END_DATE` は本来の終了日のままにしておく（履歴として）
- [ ] 終了当日に admin から一言投稿（ウェルカム投稿の次のピン留め投稿として）
  - 「β期間を終え、本ローンチへ移行しました。改めてよろしくお願いいたします」

### 5.3 終了後 1 ヶ月

- [ ] β期間中のフィードバック（フォーム回答・ヒアリングメモ）を 1 つのドキュメントに集約
- [ ] §8 の指標を計算し、評価
- [ ] v0.2 開発の優先順位を決定（`.claude/plans/foods-community-design.plan.md` §20 のリストを更新）
- [ ] 本ローンチ後の新規招待を開始（§10 参照）

## 6. トラブル対応

### 6.1 よくある問い合わせと対応

| 問い合わせ | 一次対応 | 二次対応 |
|---|---|---|
| 招待メールが届きません | 迷惑メール・プロモーションタブを確認してもらう | `/admin/invites` から再送 |
| ログインできません | 入力したメールアドレスを確認 → タイプミス・別ドメインの可能性 | パスワードリセットメール送信を案内 |
| パスワードを忘れました | ログイン画面の「パスワードを忘れた方」リンクを案内 | それでも解決しなければ admin 側で対応（手動リセット手順は §6.3） |
| 画像がアップロードできません | 画像サイズ・形式（HEIC は自動変換されるはず）を確認 → 別画像で試す | Sentry のエラー確認、不具合報告フォーム誘導 |
| PWA をホーム画面に追加できません | 機種別の手順を案内（iOS：共有→ホーム画面に追加 ／ Android：自動バナー or メニュー） | スクリーンショット付き手順を送る |
| 通知が来ません | 通知設定画面で該当タイプが ON か確認 | OS 側の通知許可も確認（MVP では Push 非対応のためアプリ内通知のみ） |
| 通知が多すぎます | 通知設定画面で OFF にしてもらう（admin_broadcast 等の必須通知は OFF 不可） | — |
| プランを変更したい | 「設定 → プラン情報」のフォームを案内 | 振込確認後、`/admin/members/:id` でプラン変更 |
| 退会したい | 「設定 → 退会申請」のフォームを案内 | フォーム受領後、`/admin/members/:id` で「退会させる」 |

#### β期間特有：「料金はかかりますか？」
- β期間中は**完全無料**であることを明確に伝える。
- β終了時に「継続される場合の料金」を改めて個別案内する旨を伝える。

### 6.2 緊急時の対応

| 事象 | 初動 | 復旧 |
|---|---|---|
| システム障害（画面が開かない） | 自分のスマホで再現確認 → Sentry 確認 → 全メンバーに LINE 等で「現在確認中です」と一報 | Claude Code に修正依頼、復旧後に admin_broadcast で報告 |
| 不適切投稿（差別・誹謗中傷・宣伝のみ） | `/admin/posts/:id` で論理削除 + 削除理由入力 | 本人に通知（自動）、必要なら個別連絡 |
| ハラスメント・メンバー間トラブル | 当事者双方に**別々にヒアリング**（公開の場では絶対にやらない） | 状況に応じて：注意のみ／一時停止／退会勧告 |
| 個人情報漏えい疑い | **即座に該当画面・該当データを admin で確認**、必要なら一時停止 | 漏洩範囲の特定 → 影響を受けるユーザーへの個別連絡 → 利用規約・プラポリに基づき対応 |
| サーバーダウン | 管理会社に連絡（連絡先は §11 に控えておく） | 復旧後、admin_broadcast で報告 |

### 6.3 admin 手動パスワードリセット手順
1. Supabase ダッシュボード → Authentication → Users で該当ユーザーを検索
2. 「Send password reset」を実行
3. 本人にメールが届くことを LINE で一報
4. それでも届かない場合は、Supabase 管理画面から一時パスワードを発行して LINE で個別連絡（ログイン後すぐ変更してもらう）

## 7. β初期メンバーへのフォローアップ

### 7.1 称号「β初期メンバー」（v0.2 で実装予定）

- 判定ロジック：`profiles.created_at` が β期間中（`<= NEXT_PUBLIC_BETA_END_DATE`）のメンバー
- 可視化：プロフィールカードに特別バッジ or 称号を表示
- MVP 段階では DB に created_at が残っているので、v0.2 でいつでも実装可能

### 7.2 継続インセンティブ案（要相談）

以下のいずれか／組み合わせで、β初期メンバーの継続を促す。

- β初期メンバー限定の**継続割引**（例：スタンダードプラン 25,000 円 → 当面 20,000 円など）
- 限定セミナーへの**優先招待**
- しのぶさんとの**1on1 オンライン相談**（月 1 回 30 分など）
- 「β初期メンバー」**称号バッジ**の付与（v0.2 実装後）

決定したインセンティブは、β終了告知と同時にお知らせする。

## 8. β期間で計測する指標

数字は Plausible・Supabase・Google フォーム回答から手動集計する。

| 指標 | 計算方法 | 目標水準（参考） |
|---|---|---|
| 招待→受諾率 | accepted 招待数 ÷ 発行招待数 | 80% 以上 |
| 受諾→初投稿までの日数（中央値） | 各メンバーの `profiles.created_at` → 初投稿日 | 7 日以内 |
| 平均投稿数／週／メンバー | 投稿総数 ÷ 週数 ÷ メンバー数 | 1 件以上 |
| データ記録継続率 | 月次データを 1 件以上入力したメンバー数 ÷ 全メンバー数 | 50% 以上 |
| 退会率 | β終了時の退会者数 ÷ 全メンバー数 | 10% 以下 |
| お問い合わせ件数・カテゴリ | フォーム回答を集計（操作方法／要望／その他） | 件数より「重複するカテゴリ」を重視 |
| 不具合報告件数・カテゴリ | フォーム回答を集計（再現性／重大度） | CRITICAL/HIGH が 0 件になる方向で改善 |
| 通知設定変更率 | 通知 OFF にしたメンバー数 ÷ 全メンバー数 | 30% 超えたらデフォルト見直し |
| DAU/MAU 概算 | Plausible のユニーク訪問者から推定 | 50% 超えれば優秀 |

### 8.1 集計のタイミング
- 週次：簡易ダッシュボード（投稿数・コメント数・アクティブ数）
- 月次／β終了時：上記指標を全て集計

### 8.2 記録の残し方
- Google スプレッドシートに「β指標」シートを作成し、週次で値を追記
- β終了後に振り返りドキュメントとして保存

## 9. β期間後の振り返り

β終了後 1 ヶ月以内に、以下を 1 つのドキュメントにまとめる。

### 9.1 振り返りドキュメントの構成

1. **数値サマリー**（§8 の全指標の実測値）
2. **ヒアリング結果のまとめ**
   - メンバーごとのコメント（匿名化）
   - 共通して挙がった改善要望
   - 共通して挙がった「良かった点」
3. **不具合報告サマリー**
   - 対応済み（v0.1 で fix）
   - v0.2 送り（理由：影響軽微 or 設計変更を伴う）
   - 未対応（理由を明記）
4. **設計改善の優先順位**
   - 高：本ローンチ前に直す
   - 中：v0.2 で対応
   - 低：将来検討
5. **本ローンチに向けた最終調整事項**
   - 価格表示
   - お問い合わせ窓口の体制（一人運用で回るか）
   - 招待ペース（一気に 50 名 or 段階的）
   - インフラ（Supabase Free → Pro 切替の検討タイミング）

### 9.2 振り返りの活用
- 振り返りドキュメントは、本ローンチ時の新規招待者への「ようこそ投稿」のネタにも使える。
- v0.2 開発計画（`.claude/plans/foods-community-design.plan.md` §20）の更新に直接反映する。

## 10. 本ローンチへの移行

### 10.1 本ローンチ前の最終確認チェックリスト

- [ ] β期間中に発見した CRITICAL/HIGH 不具合がすべて修正済み
- [ ] 価格表示（料金プラン・特商法表示）が正常
- [ ] お問い合わせ窓口の運用体制が確認できている（一人運用の負荷感）
- [ ] `NEXT_PUBLIC_BETA_MODE=false` がデプロイ済み
- [ ] 初期コンテンツ（ウェルカム投稿・利用ガイド・FAQ）が本ローンチ向けに更新済み
- [ ] 招待メールテンプレが本ローンチ用の文面に更新済み
- [ ] Plausible・Sentry の通知設定が本番運用向けに調整済み
- [ ] Supabase のプラン（Free / Pro）がメンバー規模に対して妥当
  - 100 名超えそうなら Pro（$25/月）への切替を検討

### 10.2 新規招待の開始

- 規模目標：**20〜50 名**（β初期メンバーを含む）
- 招待ペース：**1 週間あたり 5〜10 名**を推奨
  - 一気に 50 名招待すると、最初の数日に問い合わせが集中して捌けない
  - 段階的に招待することで、しのぶさん側のキャパシティに合わせられる
- 各招待は β期間と同じ手順（§4.1）で実行

### 10.3 本ローンチ後 1 ヶ月の運用

- 週次：β期間と同じ指標を継続計測
- 月末：振り返りミニ会議（一人でもよい）
- 問題が起きたら、Claude Code に即時相談
