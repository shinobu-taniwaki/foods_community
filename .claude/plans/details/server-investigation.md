# サーバー実態調査 依頼ドキュメント

**対象プロダクト**：マーケティングCampコミュニティ（MCC） MVP

**用途**：しのぶさんが管理会社（または契約中のレンタルサーバー会社）に対して問い合わせ・自己調査を行うための実務ガイド
**親ドキュメント**：[`../foods-community-design.plan.md`](../foods-community-design.plan.md) §1.2 / §15 / §21.2

## このドキュメントの使い方

このドキュメントは「読むだけ」ではなく「**埋める**」ためのものです。

1. まず §1〜§2 を読んで、調査の目的とステップを把握する
2. §3 の問い合わせ文をコピーして、管理会社宛にメール送信する（または §4 の SSH コマンドで自己調査する）
3. 回答が返ってきたら §5 の記入テンプレートを埋める
4. 埋まった内容を §6 のパターン表と照合し、§7 の次アクションを決める

最後まで埋め切れば、設計書 §15「デプロイ・インフラ」の方針を確定できる状態になります。

## 1. 調査の目的

今回の調査は、以下の 3 点を明確化するために行います。

### 1.1 新アプリの稼働環境としての適合性確認

新規アプリは **Next.js 14（Node.js 20 LTS）+ Supabase Cloud 連携** という構成です。
これを現在契約しているサーバーで動かせるかどうか、技術的に検証する必要があります。

主な確認ポイント：

- Node.js（JavaScript の実行環境）が動くか
- Nginx（Web サーバー）または同等のリバースプロキシが使えるか
- 独自ドメイン・SSL（https の暗号化通信）が利用できるか

### 1.2 データのリスクヘッジ（責任所在の明確化）

ユーザーデータが消えたり、サーバーが落ちたりした場合に、

- **どこまでが管理会社の責任か**
- **どこからがしのぶさん側の責任か**

を契約書ベースで確認します。とくに「データのバックアップが契約に含まれるか」「ハードウェア障害時に自動復旧されるか」は重要です。

### 1.3 Self-hosted Supabase 採用可否の判断

設計書では「Supabase Cloud（クラウド版の Supabase）」を採用予定ですが、サーバースペックによっては **Self-hosted Supabase（自前サーバー上で Supabase 一式を動かす）** という選択肢も取れます。

Self-hosted を取れば「データを完全に自前で保持できる」というメリットがある反面、「自分でメンテする義務が発生する」というコストもあります。サーバー実態が分からないと判断できないため、今回の調査結果で決定します。

## 2. 調査ステップの全体像

調査は 3 段階で進めます。**ステップ 1 はしのぶさん自身でできる作業**で、ステップ 3 だけが管理会社への問い合わせになります。

### ステップ 1：請求書からサービス会社を特定（しのぶさん自身で実施）

- メール / 紙の請求書ファイルを検索（キーワード例：「サーバー」「ホスティング」「VPS」「クラウド」）
- 請求会社名・サービス名・契約プラン・月額を記録
- 想定される会社例：
  - **さくらインターネット**（さくらの VPS / さくらのレンタルサーバー）
  - **GMOクラウド**（ConoHa VPS、お名前.com レンタルサーバー）
  - **IDCフロンティア**（IDCFクラウド）
  - **KDDIウェブコミュニケーションズ**（CPI）
  - **IIJ**（IIJ GIO）
  - **エックスサーバー**（XServer VPS / レンタル）
  - **ABLENET**（VPS）
  - **ロリポップ！**（GMOペパボ運営、共有レンタル）
  - **AWS / GCP / Azure**（直接契約の場合）

### ステップ 2：管理画面からのスペック確認（しのぶさん自身で実施）

- サーバー会社のマイページ（管理コンソール）にログイン
- 契約サーバーの仕様（CPU、メモリ、ディスク、OS）を確認
- バックアップオプションの有無、SSL 証明書の発行可否を確認

### ステップ 3：管理会社へ問い合わせ（必要に応じて）

- ステップ 2 で不明だった項目を §3 の問い合わせ文を使って問い合わせる
- 回答内容を §5 の記入テンプレートに反映する

### 3.1 用語の補足（管理会社の回答を理解するために）

回答に出てくる可能性のある用語をまとめます。

| 用語 | 意味 |
|---|---|
| **VPS** | Virtual Private Server。1 台の物理サーバーを仮想的に分割した専用環境 |
| **専用サーバー** | 物理サーバー 1 台を丸ごと借りる契約 |
| **共有サーバー / レンタルサーバー** | 複数の契約者で 1 台を共有。自由度は低いが安価 |
| **Node.js** | JavaScript をサーバー側で実行する環境。今回のアプリで必須 |
| **Docker** | アプリを「コンテナ」という箱に入れて配布・実行する仕組み |
| **Nginx** | Web サーバー。ブラウザからのアクセスを受け付ける入口 |
| **SSH** | Secure Shell。サーバーに安全にログインしてコマンド操作するための仕組み |
| **systemd / cron** | 「起動時に自動でアプリを立ち上げる」「毎日深夜に定期実行する」などを管理する仕組み |
| **Let's Encrypt** | 無料で SSL 証明書（https 化のための証明書）を発行できるサービス |
| **SLA** | Service Level Agreement。サービス品質保証 |
| **PITR** | Point-in-Time Recovery。任意の時点に巻き戻せるバックアップ機能 |

## 4. SSH で直接確認する場合のコマンド集

しのぶさんが管理会社経由で SSH 接続できる場合、以下のコマンドで自己調査ができます。
**ターミナル（黒い画面）にコピペして実行**するだけで結果が表示されます。

### 4.1 接続方法（参考）

```bash
# サーバーに接続する基本形
ssh ユーザー名@サーバーのIPアドレス

# 鍵認証の場合
ssh -i 秘密鍵ファイル.pem ユーザー名@サーバーのIPアドレス
```

### 4.2 調査コマンド一覧

```bash
# ─── OS とカーネル ─────────────────────────────
cat /etc/os-release
uname -a

# ─── CPU / メモリ ─────────────────────────────
nproc                          # CPU コア数
cat /proc/cpuinfo | head -30   # CPU 詳細
free -h                        # メモリ容量（GB 単位）

# ─── ディスク容量 ─────────────────────────────
df -h                          # ファイルシステムごとの空き容量
lsblk                          # ディスクとパーティション構成

# ─── Node.js があるか ─────────────────────────
which node || echo "Node.js not installed"
node --version 2>/dev/null
which npm 2>/dev/null
npm --version 2>/dev/null

# ─── Docker があるか ─────────────────────────
which docker || echo "Docker not installed"
docker --version 2>/dev/null
docker compose version 2>/dev/null

# ─── Nginx / Apache があるか ──────────────────
which nginx 2>/dev/null && nginx -v 2>&1 || echo "Nginx not installed"
which apache2 2>/dev/null && apache2 -v 2>&1 || echo "Apache not installed"
which httpd 2>/dev/null && httpd -v 2>&1 || echo "httpd not installed"

# ─── プロセス管理（systemd） ─────────────────
systemctl --version 2>/dev/null || echo "systemd not available"

# ─── 既存サービスの確認 ─────────────────────
ps aux | grep -E 'php|nginx|apache|mysql|postgres' | grep -v grep | head -20

# ─── ファイアウォール ─────────────────────────
sudo ufw status 2>/dev/null || sudo firewall-cmd --state 2>/dev/null || echo "firewall tool unknown"

# ─── ネットワーク（リッスンポート） ───────────
sudo ss -tlnp 2>/dev/null || sudo netstat -tlnp 2>/dev/null

# ─── ユーザー権限 ─────────────────────────────
whoami
id
sudo -n true 2>/dev/null && echo "sudo available" || echo "sudo requires password or unavailable"

# ─── PostgreSQL があるか ─────────────────────
which psql 2>/dev/null && psql --version || echo "PostgreSQL not installed"

# ─── 再起動権限の確認（破壊的ではない、確認のみ）─
sudo -l 2>/dev/null | head -20
```

### 4.3 コマンドの結果の見方

- **`installed`** と表示されればインストール済み
- **`not installed`** や **`command not found`** が出れば未インストール
- `free -h` の `Mem:` 行の「total」がメモリ容量
- `df -h` の `/` 行の「Size」が OS が使えるディスク容量

結果を全部コピーして、§5 のテンプレートに転記してください。

## 6. 調査結果から判断するパターン

§5 の記入テンプレートが埋まったら、以下の 4 パターンと照合して該当するものを選んでください。

### パターン A：Docker 可・メモリ 4GB 以上・SSH 可

**おすすめ構成**：**Self-hosted Supabase 採用候補**

- Supabase 一式（PostgreSQL / Auth / Storage / Realtime）をコンテナで全部自前運用
- データが完全に自前サーバーに置けるため、リスクヘッジ最強
- 責任範囲が「管理会社：ハードウェア、自分：アプリ全般」と整理しやすい
- ただし、Supabase のメンテ（バージョン更新、障害対応）を自分でやる覚悟が必要

**次の検証**：
- Self-hosted Supabase の公式 Docker 構成（docker-compose.yml）が動くか試す
- メモリ消費・ディスク消費の見積もり

### パターン B：Node.js 可・Docker 不可・メモリ 2GB 以下

**おすすめ構成**：**Supabase Cloud + 自前サーバーにバックアップ**（最も現実的）

- Next.js を自前サーバーで動かす（systemd / PM2 で常駐）
- DB / Auth / Storage は Supabase Cloud に任せる
- 毎日 `pg_dump` でバックアップを取得し、自前サーバーに保存（cron + rsync）
- 設計書 §15 の現在案そのまま

**次の検証**：
- Node.js 20 LTS のインストール手順を管理会社と確認
- systemd でのプロセス常駐運用が可能か確認

### パターン C：レンタル共有サーバー（PHP のみ動作）

**おすすめ構成**：**設計見直しが必要**

- 自前サーバーを Next.js デプロイ用には使えない
- 選択肢：
  - **A 案**：Next.js を **Vercel**（無料〜）にデプロイ。自前サーバーは別用途（バックアップ保管庫など）
  - **B 案**：別途 VPS（さくらの VPS、ConoHa VPS、エックスサーバー VPS など）を新規契約。月額 1,000〜3,000 円程度
  - **C 案**：既存サーバー契約を解約 → VPS 契約に切り替え

**次の検証**：
- 既存サーバーの解約条件を確認
- Vercel / VPS の比較見積もり

### パターン D：管理会社が契約上「OS まで」しか責任を持たない

**おすすめ構成**：**データバックアップは自分で運用**

- アプリ・データの責任は契約者側
- `pg_dump` スクリプトを自分で書く＋ cron で定期実行
- バックアップは **外部ストレージ**（AWS S3 / Cloudflare R2 / Backblaze B2 など）にも複製
  - これにより「サーバー全損 → 全データ喪失」を防げる
- 別途、災害対策として **別地域** のストレージに置くことを推奨

**次の検証**：
- 外部ストレージの選定（月額数百円〜数千円）
- バックアップスクリプトの設計

### 6.1 複合パターンの場合

実際の調査結果は複数パターンに当てはまることが多いです。
たとえば「Docker 不可（パターン B）」かつ「OS まで責任（パターン D）」のように、複合適用してください。

- §6 で決定したパターンに応じて、設計書 §15「デプロイ・インフラ」を更新

  - 環境変数の取り扱い（`.env.production` のサーバー配置）
  - バックアップデータの暗号化

- 設計書 §21.1 の **F-5：ドメイン名** をしのぶさんと確定

## 8. その他リスクと対策

調査と並行して、以下の中長期リスクも検討しておきます。

### 8.1 管理会社の廃業・サービス終了リスク

- 大手（さくら、GMO、AWS など）であればリスクは小さいが、中小ホスティング会社の場合は備えが必要
- **対策**：別管理会社への移行手順を事前に整理（バックアップフォーマットを汎用的に保つ）

### 8.2 契約更新時の値上げ

- レンタルサーバー業界は数年単位で値上げが起こる
- **対策**：年 1 回、別サービスへの比較見積もりを取る習慣をつける

### 8.3 災害・地域障害

- データセンターの所在地によっては、地震・水害・火災で物理的に消失する可能性
- **対策**：バックアップは**別地域のストレージ**に保管（東京拠点なら大阪 / 福岡など）
- Supabase Cloud + AWS S3（東京リージョン → 大阪リージョンへの複製）の組み合わせなどが現実的

### 8.4 アカウント乗っ取りリスク

- 管理会社のマイページや SSH 鍵が漏れると、サーバー一式を乗っ取られる
- **対策**：
  - 管理会社マイページに **二段階認証（2FA）** を必ず有効化
  - SSH は **鍵認証のみ**（パスワード認証は無効化）
  - 鍵ファイルは パスフレーズ付き で生成

### 8.5 契約者（しのぶさん）に何かあった場合の引き継ぎ

- 個人運営の場合、契約者が事故・病気で対応不可能になるリスク
- **対策**：契約情報・パスワード・鍵を**信頼できる第二者**または**パスワードマネージャー経由**で承継可能にしておく
